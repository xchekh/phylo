# Оценка резервов в аудиторских компаниях — инструменты и методы

## Что такое резервы под кредитные потери

Резервы (Loan Loss Provisions / Expected Credit Loss, ECL) — сумма, которую банк признаёт в P&L как ожидаемые потери по кредитному портфелю. Это один из самых субъективных и рискованных участков банковской отчётности с точки зрения аудита.

**Почему это зона высокого аудиторского риска:**
- Оценка требует суждений о будущем (вероятность дефолта, ожидаемые потери)
- Менеджмент заинтересован в занижении резервов → завышение прибыли и капитала
- Расчёты основаны на внутренних моделях, которые трудно верифицировать
- Объём резервов напрямую влияет на достаточность капитала (Н1.0)

## Регуляторная и стандартная база

| Документ | Область применения |
|----------|--------------------|
| **IFRS 9** (с 2018 г.) | МСФО — модель ECL (ожидаемых кредитных потерь) |
| **МСФО 39** (до 2018 г.) | Модель понесённых потерь (incurred loss) — устарела |
| **Положение ЦБ № 590-П** | РСБУ — резервы по ссудам для российских банков |
| **Положение ЦБ № 605-П** | РСБУ — резервы по прочим активам |
| **ISA 540** | Стандарт аудита оценочных значений (Auditing Accounting Estimates) |
| **ISA 330** | Ответные процедуры на оцениваемые риски |
| **ISA 315** | Оценка рисков существенного искажения |

## IFRS 9 — Трёхстадийная модель ECL

Вся логика аудита резервов строится вокруг трёх стадий IFRS 9:

```
Стадия 1 (Stage 1)
  Кредит выдан, кредитный риск не увеличился существенно
  Резерв = ECL за 12 месяцев (12-month ECL)
  Признание процентного дохода: от валовой балансовой стоимости

Стадия 2 (Stage 2)
  Существенное увеличение кредитного риска (SICR) с момента выдачи
  Резерв = ECL за весь срок жизни актива (Lifetime ECL)
  Признание процентного дохода: от валовой балансовой стоимости

Стадия 3 (Stage 3 = NPL)
  Кредит обесценен (default, просрочка 90+ дней или признаки)
  Резерв = Lifetime ECL (обычно очень высокий)
  Признание процентного дохода: от чистой балансовой стоимости (нетто)
```

### Формула ECL
```
ECL = PD × LGD × EAD × DF

PD  — Probability of Default (вероятность дефолта)
LGD — Loss Given Default (потери при дефолте = 1 − Recovery Rate)
EAD — Exposure at Default (сумма под риском на момент дефолта)
DF  — Discount Factor (дисконтирование к отчётной дате)
```

## Роль аудитора при проверке резервов

Аудитор **не пересчитывает резервы с нуля** — он оценивает, насколько методология и расчёты менеджмента разумны и соответствуют стандартам.

**Три подхода аудитора по ISA 540:**
1. **Проверка процесса** — оцениваем, как менеджмент строит оценку, адекватны ли контроли
2. **Разработка независимой оценки** — аудитор сам оценивает ECL и сравнивает с управленческой
3. **Ретроспективный анализ** — проверяем, насколько прошлые оценки менеджмента оказались точными (back-testing)

## Инструменты аудиторской оценки резервов

---

### 1. Независимое моделирование ECL (Independent ECL Model)

Аудиторская фирма строит собственную модель расчёта ECL — параллельно модели банка.

**Как работает:**
- Выгружают данные кредитного портфеля из систем банка
- Применяют собственные PD/LGD/EAD оценки (из рейтинговых агентств, отраслевых баз, регуляторных данных)
- Сравнивают итоговый ECL с тем, что признал банк
- Разница → аудиторский диапазон (auditor's range)

**Инструменты:**
- **Python / R** — статистические модели, регрессии на исторических данных дефолтов
- **Excel / VBA** — для сегментных расчётов небольшого портфеля
- **SAS** — стандарт в крупных банках и аудите
- **Специализированные платформы:** Moody's RiskCalc, S&P RiskGauge, внутренние инструменты Big4

---

### 2. Тестирование данных и систем (Data & IT Testing)

Резервы считаются из данных — если данные неверны, резерв неверен.

**Что проверяют:**
- **Completeness** — все кредиты попали в расчёт (выгрузка из АБС полная)
- **Accuracy** — суммы, даты, ставки, просрочка загружены корректно
- **Cut-off** — кредиты отражены в правильном периоде
- **Classification** — правильно ли разнесены кредиты по стадиям (1/2/3)

**Инструменты:**
- **ACL Analytics / Galvanize (ACL)** — аудиторская СУБД для анализа больших массивов данных, SQL-запросы, поиск дублей, выбросов
- **IDEA (Interactive Data Extraction and Analysis)** — аналог ACL, популярен в KPMG / Deloitte
- **Python (pandas, sqlalchemy)** — кастомные скрипты для сверки данных
- **SQL** — прямые запросы к банковским системам (АБС: ЦФТ, Банкир, Misys)
- **Power BI / Tableau** — визуализация портфельных данных и аномалий

---

### 3. Сегментация портфеля и выборочная проверка (Sampling)

Портфель делится на сегменты, по каждому применяется свой подход.

**Стратификация:**
```
Крупные кредиты (Individual Assessment)
  → Индивидуальная проверка каждого (100% покрытие)
  → Анализ финотчётности заёмщика, обеспечения, переписки

Однородные группы (Collective Assessment / Portfolio Approach)
  → Статистические модели ECL
  → Ипотека, потребкредиты, автокредиты — пул однородных позиций
```

**Методы выборки:**
- **Monetary Unit Sampling (MUS)** — каждый рубль имеет равную вероятность попасть в выборку → смещение в сторону крупных кредитов
- **Stratified Sampling** — портфель разбивается на страты (по размеру, отрасли, просрочке), из каждой — своя выборка
- **Judgmental Sampling** — аудитор отбирает вручную: самые крупные, реструктурированные, аффилированные

**Инструменты:**
- **EY Canvas, Deloitte Argus, KPMG Clara, PwC Aura** — внутренние аудиторские платформы Big4 со встроенными модулями выборки
- **CaseWare** — документирование аудита и выборок
- **TeamMate Analytics** — модуль для статистических выборок

---

### 4. Анализ SICR — существенного увеличения кредитного риска

SICR — ключевой триггер перевода из Стадии 1 в Стадию 2. Банки часто занижают SICR, чтобы не досоздавать резервы.

**Что проверяет аудитор:**
- Какие индикаторы банк использует для определения SICR (просрочка, ухудшение рейтинга, watchlist, финансовые ковенанты)
- Применяется ли «треугольник» миграций (vintage analysis)
- Есть ли «backstop» — 30-дневная просрочка автоматически переводит в Стадию 2

**Инструменты:**
- **Vintage / Migration Analysis** — матрица переходов между рейтингами / стадиями за N периодов
- **Logistic Regression** — модель предсказания перехода в дефолт
- **Survival Analysis** — время до дефолта (применяется для PD-калибровки)

---

### 5. Верификация PD, LGD, EAD (Model Validation)

**PD (Probability of Default):**
```
Источники для сравнения:
- Исторические данные дефолтов по портфелю банка (internal data)
- Рейтинговые агентства: S&P, Moody's, АКРА — публичные таблицы дефолтности
- ЦБ РФ: статистика по сегментам кредитования
- Moody's Analytics, Bloomberg — отраслевые PD benchmarks
```

**LGD (Loss Given Default):**
```
LGD = 1 − Recovery Rate

Зависит от:
- Наличие и качество залога (недвижимость, оборудование, поручительство)
- Очерёдность кредитора (senior, subordinated)
- Отрасль заёмщика
- Текущая макросреда (в кризис Recovery Rate падает)

Проверка аудитором:
- Анализ исторических взысканий по дефолтам (workout data)
- Независимая оценка залога (привлечение оценщика)
- Сравнение с рыночными LGD для сопоставимых заёмщиков
```

**EAD (Exposure at Default):**
```
EAD = Текущий баланс + CCF × Незадействованный лимит

CCF (Credit Conversion Factor) — какую долю незадействованного лимита
  заёмщик выберет к моменту дефолта.

Аудитор проверяет: корректность CCF, правильность учёта кредитных линий,
  гарантий, аккредитивов в расчёте EAD.
```

### 6. Макроэкономические прогнозы (Forward-Looking Information)

IFRS 9 требует учёта «future economic conditions» в расчёте ECL. Это самая субъективная часть.

**Что проверяет аудитор:**
- Какие макропеременные используются: ВВП, ключевая ставка, цена нефти, курс рубля, безработица
- Сколько сценариев построено (min 3: base / optimistic / pessimistic)
- Как присваиваются вероятности сценариям
- Откуда берутся прогнозы (ЦБ РФ, МВФ, Bloomberg Consensus)
- Как далеко уходит «reasonable and supportable» период прогноза (обычно 1–3 года)

**Инструменты:**
- **Bloomberg / Refinitiv** — макропрогнозы, консенсус-оценки
- **Oxford Economics, Moody's Analytics** — коммерческие макромодели, используемые Big4 для независимых прогнозов
- **Корреляционный анализ** — связь между макропеременными и PD / LGD по историческим данным
- **Scenario Weighting Analysis** — проверка весов сценариев через Back-testing

---

### 7. Back-testing и ретроспективный анализ

Аудитор проверяет, насколько прошлые оценки менеджмента были точны.

```
Back-testing:
  ECL(t-1), рассчитанный год назад → сравниваем с фактическими потерями за год t
  Если ECL систематически занижен → признак предвзятости менеджмента

Vintage Analysis (матрица урожайности):
  Разбивка портфеля по году выдачи
  Для каждого «урожая» смотрим: какой % перешёл в дефолт через 1/2/3 года
  Позволяет калибровать PD и выявлять ухудшение стандартов андеррайтинга
```

**Инструменты:**
- **Матрицы миграций (Transition Matrices)** — переходы между рейтингами / стадиями
- **Hosmer-Lemeshow Test** — статистический тест калибровки PD-модели
- **Gini / AUC (Area Under ROC Curve)** — оценка дискриминационной силы PD-модели
- **Python (sklearn, statsmodels)** — для статистических тестов

---

### 8. Анализ обеспечения (Collateral Assessment)

LGD напрямую зависит от качества залога. Завышенная стоимость залога → заниженный LGD → заниженные резервы.

**Что проверяет аудитор:**
- Актуальность независимой оценки (не старше 1–3 лет в зависимости от типа)
- Качество оценщика (аккредитован ли банком / аудитором)
- Применён ли дисконт ликвидности (forced sale discount)
- Учтены ли расходы на взыскание (legal costs, foreclosure costs)
- Для недвижимости — рыночная vs ипотечная стоимость

**Инструменты:**
- **Привлечение эксперта аудитора (Auditor's Expert)** по МСФО ISA 620 — независимый оценщик
- **Индексы рынка недвижимости** (Росреестр, CBRE, JLL, Savills) — для массовой переоценки ипотечного портфеля
- **Regression-based AVM (Automated Valuation Model)** — массовая оценка жилой недвижимости через рыночные данные

---

### 9. Анализ реструктурированных кредитов (Forbearance)

Реструктуризация — снижение ставки, пролонгация срока, кредитные каникулы — часто скрывает проблемные кредиты.

**Красные флаги для аудитора:**
- Многократные реструктуризации одного заёмщика
- Кредит реструктурирован, но остался в Стадии 1
- Снижение ставки без изменения стадии → потенциальное занижение ECL
- Кредиты с «evergreening» — банк рефинансирует просроченный кредит, чтобы не признавать дефолт

**Инструменты:**
- **Forbearance Register** — проверка полноты реестра реструктурированных кредитов
- **Анализ кредитной истории** (credit file review) — изучение оригинальных документов по выбранным кредитам
- **Timeline Analysis** — ретроспектива изменений условий кредита

---

### 10. Процедуры по крупным корпоративным кредитам (Individual Impairment)

Для крупных заёмщиков (обычно > 1–5% капитала банка) — индивидуальная проверка.

**Что запрашивает аудитор:**
- Финансовая отчётность заёмщика (МСФО / РСБУ) за 3 года
- Кредитный файл: меморандум, решение кредитного комитета
- Управленческая отчётность, бюджет, бизнес-план
- Отраслевые данные, новости, судебные дела
- Аудиторское заключение по отчётности заёмщика

**Анализ платёжеспособности заёмщика:**
```
Debt / EBITDA → долговая нагрузка
EBITDA / Проценты → покрытие процентов
FCF → способность обслуживать и погашать долг
Ликвидность → краткосрочные обязательства vs оборотные активы
```

**Инструменты:**
- **DCF / Discounted Cash Flow** — оценка стоимости бизнеса для сравнения с долгом (если долг > стоимость бизнеса → полный дефолт)
- **Waterfall Analysis** — в случае дефолта: что достанется кредитору с учётом залогов, очерёдности, расходов на взыскание
- **Stress Testing заёмщика** — сценарии: падение выручки -20/-30%, рост ставок, девальвация

## Документирование и рабочие бумаги аудитора

Для каждого вывода по резервам аудитор оформляет рабочие бумаги:

```
1. Оценка риска (Risk Assessment)
   — Почему резервы — область высокого риска
   — Ключевые суждения менеджмента

2. Тест средств контроля (Controls Testing)
   — Оцениваем: есть ли надёжный контроль над расчётом резервов в банке
   — Кредитный комитет, независимая валидация моделей, IT-контроли

3. Процедуры по существу (Substantive Procedures)
   — Независимая оценка ECL
   — Выборочная проверка кредитных файлов
   — Верификация данных

4. Аудиторский диапазон (Auditor's Range)
   — Нижняя и верхняя граница разумного ECL по мнению аудитора
   — Если резерв банка вне диапазона → корректировка или оговорка

5. Раскрытие информации (Disclosures)
   — Проверяем, что банк раскрыл всё требуемое IFRS 7:
     PD-распределение, LGD, сценарии, чувствительность
```

## Типичные искажения и как их обнаруживают

| Схема | Как выявляют |
|-------|-------------|
| Занижение PD через оптимистичные внутренние рейтинги | Сравнение с внешними рейтингами / рыночными CDS |
| Завышение стоимости залога → заниженный LGD | Независимая оценка аудиторского эксперта |
| Необоснованный вес оптимистичного сценария | Back-testing исторических прогнозов менеджмента |
| «Вечнозелёные» кредиты — рефинансирование просрочки | Анализ timeline изменений условий кредита |
| Неправомерное оставление в Стадии 1 (SICR не признан) | Проверка индикаторов SICR, backstop-тест |
| Занижение EAD по кредитным линиям | Проверка CCF, анализ выборки по кредитным линиям |
| Отсутствие Forward-Looking Adjustments в кризис | Сравнение с консенсус-прогнозами, отраслевой практикой |

## Быстрая схема аудита резервов

```
Планирование
  └── Оценка риска: почему резервы — ключевая область риска

Тест контролей
  └── Кредитный комитет, независимая валидация моделей, IT-контроли

Процедуры по существу
  ├── Данные: ACL / IDEA → полнота и точность портфельных данных
  ├── Классификация по стадиям → SICR анализ
  ├── Коллективные резервы → независимая ECL модель (Python / SAS)
  │     ├── PD верификация (vs рейтинговые агентства, ЦБ)
  │     ├── LGD верификация (независимая оценка залогов)
  │     ├── EAD / CCF проверка
  │     └── Forward-looking: сценарии, веса, источники прогнозов
  └── Индивидуальные резервы (крупные кредиты)
        ├── Анализ фин. отчётности заёмщика
        ├── DCF / Waterfall анализ
        └── Back-testing: прошлые оценки vs факт

Формирование вывода
  ├── Аудиторский диапазон ECL
  ├── Разница с резервом банка → в пределах допустимого или нет
  └── Проверка раскрытий (IFRS 7)
```
